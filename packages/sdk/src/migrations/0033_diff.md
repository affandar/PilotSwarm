# CMS Migration 0033 Diff

Migration file: `cms-migrations.ts` — `migration_0033_grant_share_create_only_grantee`

## Problem (adversarial review, MEDIUM)

`cms_grant_session_share` resolved the grantee by calling `cms_register_user`,
which does **update-on-sighting**: a non-null `email`/`display_name` passed
with the grant **overwrote the stored identity** of an existing user row.

A grant only requires the caller to own the (throwaway) session being shared,
and the grant API/MCP accept caller-supplied `email`/`display_name` for an
arbitrary target `(provider, subject)`. So any authenticated user could:

1. Enumerate users via `list_known_users` → harvest every `(provider, subject)`.
2. `grant_session_share(myOwnSession, {provider, subject: victim, email:"evil@x",
   displayName:"IT Admin"}, "read")` for each victim.
3. Rewrite how every victim appears in **session lists (owner initials)**, the
   **member directory**, and **share dialogs**.

Not an access bypass, but identity tampering / impersonation in a product
built on trustworthy attribution.

## Change

`cms_grant_session_share` (same 8-arg signature — `CREATE OR REPLACE`,
rolling-deploy safe) now resolves-or-**creates** the grantee keyed ONLY on
`(provider, subject)` with **NULL display fields**:

```sql
INSERT INTO users (provider, subject) VALUES (...) ON CONFLICT DO NOTHING;
SELECT user_id ... ;
```

- An existing (sighted or placeholder) row is **never touched** — the overwrite
  vector is gone.
- A brand-new grantee placeholder has `display_name = NULL`, so it is correctly
  excluded from `cms_list_users` (which filters `display_name IS NOT NULL`)
  until that person actually signs in — the inject-a-fake-named-entry vector is
  also gone. This restores migration 0031's stated invariant that grants carry
  no display_name.
- Names/emails enter the directory ONLY from a principal's own sightings
  (login / session-create → `cms_register_user`). Email-keyed placeholders
  still fold into the real principal via 0032 adoption on first sign-in.

`p_email`/`p_display_name` are retained in the signature for stability but are
no longer written.

## Consumers

No client change required. `grantSessionShare` (runtime.js, MCP
`grant_session_share`, both transports) may still pass email/display_name; they
are now ignored for directory writes. Server-side only, applies fleet-wide as
soon as the migration runs.
