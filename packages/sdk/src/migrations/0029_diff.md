# CMS Migration 0029 Diff

Migration file: `cms-migrations.ts` — `migration_0029_session_visibility_shares`

The security-model schema (docs/proposals/user-admin-security-model.md):
session visibility + targeted shares, denormalized tree roots so viewer
filtering is one join, and an authz audit trail. Access always resolves at
the session-tree ROOT — a child's own `visibility` is not consulted.

## Table Changes

- `sessions`: adds `visibility TEXT NOT NULL DEFAULT 'private'`
  (`private | shared_read | shared_write`; meaningful on root sessions only)
  and `root_session_id TEXT` (stamped at create; backfilled here by a
  depth-capped recursive parent walk, orphans fall back to self-rooted).
- New `session_shares (session_id, user_id) PK`: targeted per-user grants
  (`access: read | write`) keyed on the ROOT session id, with `granted_by`
  and `granted_at`. `session_id` cascades on session delete; `user_id`
  references `users`.
- New `authz_audit`: append-only record of denials, break-glass reads, and
  share/visibility changes (`actor_*`, `action`, `session_id`, `target`,
  `decision`, `reason`, `details JSONB`).

## New Indexes

- `idx_<schema>_sessions_root` on `sessions(root_session_id)`
- `idx_<schema>_session_shares_user` on `session_shares(user_id)`
- `idx_<schema>_authz_audit_session` on `authz_audit(session_id, occurred_at DESC)`
- `idx_<schema>_authz_audit_occurred` on `authz_audit(occurred_at DESC)`

## New Stored Procedures

- `cms_resolve_root_session(TEXT) RETURNS TEXT` — live session's root id,
  self-fallback when the recorded root row is hard-gone, NULL when the
  session is missing/soft-deleted.
- `cms_set_session_visibility(TEXT, TEXT)` — validates the enum, resolves
  the root, refuses system trees.
- `cms_grant_session_share(session_id, provider, subject, email,
  display_name, access, granted_by_provider, granted_by_subject)` —
  registers the grantee via `cms_register_user`, upserts the share on the
  root (`ON CONFLICT … DO UPDATE access/granted_by/granted_at`), refuses
  system trees.
- `cms_revoke_session_share(TEXT, TEXT, TEXT)`.
- `cms_list_session_shares(TEXT)` — grants on the root with grantee profile
  fields and grantor display name.
- `cms_get_session_access(session_id, viewer_provider, viewer_subject)` —
  one round-trip access snapshot: root id, root's `is_system`/`visibility`/
  owner, `viewer_is_owner`, `viewer_share_access`. Zero rows = session
  missing or soft-deleted. Reports facts only; the caller combines with the
  caller's role (admin bypass etc.).
- `cms_record_authz_audit(...)` / `cms_list_authz_audit(limit, session_id)`
  (list is newest-first, limit clamped to 500). When `session_id` is given it
  matches the whole session TREE — rows recorded against child sessions of the
  resolved root are included, so an owner querying their root sees break-glass/
  deny activity across the tree, not just the exact id.
- `cms_list_sessions_visible(viewer_provider, viewer_subject,
  system_visible)` — non-paged viewer-scoped listing (same shape as
  `cms_list_sessions`).

## Changed Stored Procedures

- `cms_create_session` — new 10-arg overload (0026's 9 args +
  `p_visibility`). Stamps `root_session_id` (parent's root, else self) and
  validated `visibility` at insert and on the deleted-resurrect conflict
  path. **No parameter defaults** — defaults would make 9-arg calls
  ambiguous against the retained 0026 overload (kept for older writers
  during rolling deploys).
- `cms_get_session(TEXT)` (DROP + CREATE): `RETURNS TABLE` appends
  `visibility`, `root_session_id` after `splash_mobile`.
- `cms_list_sessions()` / `cms_list_group_sessions(TEXT)` (DROP + CREATE,
  recreated together — the group variant is `SELECT *` over the base so
  shapes must stay identical): same two appended columns.
  Viewer comparisons in the list procs `BTRIM` the viewer provider/subject to
  match `cms_get_session_access` (defense-in-depth consistency).
- `cms_list_sessions_page` (DROP + CREATE): same two appended columns, plus
  three new trailing parameters `p_viewer_provider TEXT DEFAULT NULL`,
  `p_viewer_subject TEXT DEFAULT NULL`, `p_viewer_system_visible BOOL
  DEFAULT TRUE`. `p_viewer_provider IS NULL` (the old 4-arg call shape) is
  unfiltered — existing callers see today's behavior. A non-NULL viewer
  restricts rows to: system sessions (when `p_viewer_system_visible`), trees
  whose ROOT the viewer owns, roots with `shared_read`/`shared_write`
  visibility, and roots with a targeted share for the viewer. Defaults are
  unambiguous because the old 4-param signature is dropped in the same
  migration.

## Direct Table Access

None added. `session_shares` and `authz_audit` are reachable only through
the procs above, per CMS convention.

## TypeScript Callers

- `SessionRow` gains `visibility` and `rootSessionId`; `rowToSessionRow`
  maps them with `'private'` / self-id fallbacks so pre-0029 rows read
  safely.
- `PgSessionCatalog.createSession` gains `visibility?` and probes for the
  10-arg overload (same pattern as the 0026 `splashMobile` probe).
- New catalog methods: `setSessionVisibility`, `grantSessionShare`,
  `revokeSessionShare`, `listSessionShares`, `getSessionAccess`,
  `recordAuthzAudit`, `listAuthzAudit`, `listSessionsVisible`;
  `listSessionsPage` gains viewer options.
- Enforcement consumers: `packages/app/web/runtime.js` (predicate at the
  dispatch chokepoint), `packages/app/web/api/ws.js` (subscribe gate),
  management-client share ops.

## Behavioral Notes

- Existing rows get `visibility = 'private'` from the column default. With
  `AUTHZ_ENFORCE_OWNERSHIP=false` (the dark-launch default) nothing changes
  at runtime. Deployments that want pre-0029 sessions deployment-shared
  before flipping enforcement can bulk-set root sessions to
  `shared_write` (documented in the proposal's migration notes).
- System trees: not sharable, visibility pinned private; user-facing
  read access to them is a runtime policy decision
  (`SESSIONS_SYSTEM_VISIBILITY`), not a schema property.

## Hardened Execution Shape (post-first-deploy)

The first production run (pilotswarm-aks, 2026-07-15, 72 rows) executed the
original single-transaction shape safely, but adversarial review flagged that
the shape holds the ALTER's `ACCESS EXCLUSIVE` lock on `sessions` through the
full-table backfill and a non-CONCURRENT index build — a fleet-wide freeze on
any large, busy deployment migrating online. The migration is now a
`steps` migration (see `MigrationEntry.steps` in `pg-migrator.ts`), with an
identical end state:

1. Both `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` commit alone under
   `SET LOCAL lock_timeout = '5s'` (metadata-only; a blocked ALTER fails fast
   and the worker's CMS-init retry re-attempts instead of queueing the fleet).
2. The `root_session_id` backfill runs level-by-level in a DO block that
   COMMITs between passes (PG 11+): brief row locks only, never a table lock.
   Cycles self-exhaust; orphans and cycle members fall back to self-rooted
   (the pass cap is 128, superseding the old depth-64 CTE cap).
3. `idx_<schema>_sessions_root` builds with `CREATE INDEX CONCURRENTLY`,
   preceded by an invalid-leftover sweep (a died CIC leaves an INVALID index
   that `IF NOT EXISTS` would otherwise treat as present).
4. The new tables and all function DDL remain one atomic step, so the
   DROP+CREATE proc swaps expose no missing-function window.

Every step is idempotent; a mid-way failure leaves the version unrecorded and
the next run re-executes all steps. Already-migrated deployments are
unaffected (version-keyed skip).
