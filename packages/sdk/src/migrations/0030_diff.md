# CMS Migration 0030 Diff

Migration file: `cms-migrations.ts` — `migration_0030_register_user_update_on_sighting`

Flips `cms_register_user` from first-seen-write-wins to **update-on-sighting**
for the display fields (`email`, `display_name`).

## Motivation

A `users` row is created the first time a principal is *seen*. With the
security model, a share grant (`cms_grant_session_share`) is a sighting that
carries only `(provider, subject)` — the granter rarely knows the grantee's
email/display name. Under first-seen-write-wins that null-display row was
frozen, so when the grantee later signed in and created sessions, the
owner-initials UI rendered `?` for their own sessions (the join found a user
row with null display fields). This is the "update-on-sighting" amendment
called out in the user-lifecycle section of
`docs/proposals/user-admin-security-model.md`.

## Changed Stored Procedures

- `cms_register_user(provider, subject, email, display_name)` — `CREATE OR
  REPLACE`, **same signature** (returns `BIGINT`), so no drop needed. The
  `ON CONFLICT` clause changes from `DO NOTHING` to `DO UPDATE` with
  `COALESCE(EXCLUDED.<field>, users.<field>)`:
  - a sighting carrying a non-null `email`/`display_name` (a real login /
    session create) **refreshes** the stored value;
  - a sighting carrying nulls (a share grant) leaves existing values
    **untouched**;
  - the identity key `(provider, subject)` is never changed;
  - `updated_at` bumps only when a field actually changes (guarded by a
    `WHERE … IS DISTINCT FROM …` on the conflict update).

## Tables / Indexes

None.

## TypeScript Callers

None — the `cms_register_user` signature and `SessionRow.owner` shape are
unchanged. Behavior only: owner display fields now reflect the latest sighting
that carried them, so `?` initials disappear once an owner is seen with real
profile data (e.g. their first portal session create after a prior share
grant).

## Behavioral Note

Existing rows with null display fields are backfilled the next time their
principal is sighted with real values — no data migration needed; the join
resolves the refreshed profile for all of that owner's sessions at once.
