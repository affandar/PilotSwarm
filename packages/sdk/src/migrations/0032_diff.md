# CMS Migration 0032 Diff

Migration file: `cms-migrations.ts` — `migration_0032_adopt_email_keyed_grants`

## Problem

The share UX (portal Modify dialog, `grant_session_share` MCP tool) is a free
text box: the known-users directory is only an autocomplete **helper**, and a
grant may target someone who has **never signed in**. The grant path stores
what it was typed, creating a placeholder user row keyed
`(provider, subject = typed text)` — usually an email.

For providers with opaque subjects (Entra: subject = OID GUID), the real
principal that shows up on first sign-in is keyed by OID, a *different* row —
so the email-keyed grant would silently never bind.

## Change

`cms_register_user` (same 4-arg signature as 0030 — `CREATE OR REPLACE`,
rolling-deploy safe) now performs **adoption on sighting**: whenever a
sighting carries an email, any placeholder rows keyed
`(same provider, subject = that email, different user_id)` are folded into
the real user:

- `session_shares` rows move to the real `user_id`; where both users hold a
  grant on the same session, the stronger access (`write`) wins.
- Residual `session_shares.granted_by`, `session_owners.user_id`, and
  `session_group_owners.user_id` references re-point (defensive — a
  placeholder never signs in, so it should own/grant nothing).
- The placeholder user row is deleted (it never satisfies `cms_list_users`'s
  `display_name IS NOT NULL` filter, so the directory was already clean).

The subject match is case-insensitive; the users table is a small directory,
so the scan is cheap on every sighting.

## Consumers

No client-side changes required — behavior is entirely server-side, so it
applies fleet-wide as soon as the migration runs, including to messages from
older pods.
